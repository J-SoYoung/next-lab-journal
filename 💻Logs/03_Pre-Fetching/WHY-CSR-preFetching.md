# Next.js의 SSR, CSR, 그리고 Pre-Fetching 이해하기

## ❓ Q1.  
Next.js로 프로젝트를 시작할 때의 첫 로드는 SSR/SSG로 사전 렌더링된다.  
그런데 Next.js는 사전 렌더링만으로 서비스를 운영하기에 무리가 있어서  
CSR을 따로 구현하려는 걸까? (pre-fetch 같은 기능을 보면 그렇게 느껴짐)

### 💡 A1. 
**Next.js는 SSR이 무거워서 CSR을 사용하는 게 아니라**,  
**사용자 경험(UX)을 최적화하기 위해 SSR과 CSR을 함께 사용하는 하이브리드 구조**를 갖춘것이다.<br/>

SSR(또는 SSG)은 첫 화면을 빠르게 보여주고 SEO에 유리하지만,  
모든 페이지 전환을 SSR로 처리하면 매번 서버에 요청해야 해서 느려질 수 있다.  
그래서 Next.js는 **“첫 페이지는 SSR, 이후 전환은 CSR”** 이라는 전략을 사용한다.

| 시점 | 렌더링 전략 | 목적 |
|------|--------------|------|
| **초기 진입 (첫 로드)** | SSR / SSG | SEO, 초기 표시 속도(LCP 개선) |
| **이후 이동 (네비게이션)** | CSR | 부드러운 UX, 빠른 페이지 전환 |

=> Next.js는 **SSR의 장점 + CSR의 장점을 모두 결합한 프레임워크**다.

------

## ❓ Q2.  
그럼 Pre-Fetching은 Next.js의 사전 렌더링이  
전체 JS 번들을 한 번에 가져오지 않기 때문에,  
“페이지 단위 JS 로드의 단점을 보완하기 위해” 생긴 기능인가?

### 💡 A2.  
맞다!!! 👏  <br/>
Next.js는 **페이지 단위 코드 분할(Code Splitting)** 을 수행하기 때문에, 각 페이지마다 별도의 JS 번들이 만들어진다. 이는 초기 로드 속도를 빠르게 하지만,  
다음 페이지로 이동할 때 JS가 없으면 다시 네트워크 요청이 발생하기 때문에 Next.js는 **Pre-fetching**을 도입한 것이다.

사용자가 페이지를 보고 있을 때,
페이지에서 **이동 가능한 링크들의 JS 번들을 미리 백그라운드에서 다운로드**해 두고, **CSR처럼 즉시 전환되는 UX**를 제공한다.

| 개념 | 설명 |
|------|------|
| **코드 분할 (Code Splitting)** | 페이지별로 JS를 쪼개서 초기 로딩 속도 개선 |
| **Pre-Fetching** | 다음 이동이 예상되는 페이지 JS를 미리 받아 빠른 전환 지원 |
| **결과** | SSR의 초기 속도 + CSR의 빠른 전환 = 하이브리드 UX 완성 |

------
## 정리
- Next.js는 SSR이 “무겁기 때문”이 아니라 **UX 향상을 위해 SSR과 CSR을 혼합**하여 초기 로드는 SSR/SSG를 사용하고 이후 전환은 CSR처럼 동작한다.
- **Pre-Fetching**은 “페이지 단위 코드 분할”로 인한 전환 지연을 **CSR 수준으로 부드럽게 보완하기 위한 기술**이다.
- 결과적으로 Next.js는 SSR의 안정성과 CSR의 유연성을 동시에 갖춘 **하이브리드 렌더링 프레임워크**이다.
